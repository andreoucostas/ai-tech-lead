## WSD-006: guard.sh warns loudly instead of blocking when no JSON parser exists
- **Date**: 2026-07-01
- **Status**: Accepted (shipped in v0.23.3)
- **Decision**: with neither `jq` nor `python3` on PATH, `guard.sh` allows the write but prints a
  `write-guard INACTIVE` warning to stderr; docs state the parser dependency honestly.
- **Context**: the header claimed secrets "fail closed", but the no-parser path allowed everything
  silently — a silent security-floor outage. Blocking outright would brick every write on minimal
  boxes.
- **Alternatives**: fail closed (block all writes) — rejected as unusable; stay silent — rejected as
  dishonest.
- **Consequences**: the floor's absence is observable in hook logs; installing `jq` restores it.

## WSD-009: post-write `timeoutSec` unified to 120 across both repos; hooks.json is now gated
- **Date**: 2026-07-04
- **Status**: Accepted (shipped in v0.25.2; backlog B-05)
- **Decision**: set the Copilot `postToolUse` `post-write` `timeoutSec` to **120** in both repos'
  `.github/hooks/hooks.json` (was dotnet 120 / angular 60), and gate `hooks.json` in
  `check-lockstep.ps1` via a structured registration compare (`event|bash-script|timeoutSec` set
  equality, with `audit-trail` the one declared dotnet-only reg until B-14 ports it to Angular).
- **Context**: `hooks.json` is security-relevant (it wires guard/route-prompt/post-write/session-start
  on Copilot) yet was gated by nothing, and the timeout had already silently drifted. A higher ceiling
  is strictly safer: the hook self-throttles on frequency, so the timeout only bounds a single build/
  type-check — too low a value kills a cold `dotnet build`/`npx tsc` mid-run and drops its feedback.
- **Alternatives**: document the drift as deliberately stack-flavoured (dotnet slower than tsc) —
  rejected: the ceiling has no downside when generous, and one value is simpler to gate and reason about.
- **Consequences**: a self-test (`.claude/hooks/tests/CheckLockstep.Tests.ps1`) plants a timeout drift
  and asserts the gate catches it. When B-14 ports `audit-trail` to Angular, drop the `audit-trail`
  exception here and move `hooks.json` toward the IDENTICAL class.

## WSD-011: Token policy is the v0.8.0 static per-agent model tiering; no routing/budget framework
- **Date**: 2026-07-05
- **Status**: Accepted (records existing practice; no behavioral change)
- **Decision**: the framework's and meta layer's token-cost management is **static per-agent model
  tiering as shipped in v0.8.0** — no dynamic router, no token-budget framework, no additional
  multi-agent orchestration. The tiering rubric is the v0.8.0 one (CHANGELOG "Token economy →
  Model routing", both repos): **recurring, pattern-based work → `model: haiku`**
  (`convention-check`, `bloat-radar`, `debt-radar`); **high-stakes or one-time-high-leverage →
  `model: inherit`** (`security-auditor`, `solid-check`, `test-critic`, `bootstrap-pass`). This
  entry records and forward-references that rubric for future agents; it does not revise it.
- **Context**: prompted by the maintainer's question whether the framework should adopt a
  token-usage-management framework (cheap-model routing, agent "teams"). Primary-source consensus
  supports the existing shape and rejects heavier machinery: Anthropic's *Building Effective
  Agents* (prefer the simplest architecture that works) and multi-agent research write-up
  (multi-agent ≈15× chat tokens; pays off for parallelizable research, not context-coupled
  coding; the "token spend explains ~80% of performance variance" figure is from browsing-style
  research evals, not coding); Cognition's *Don't Build Multi-Agents* (coding is context-coupled
  — keep the main loop single-threaded on a strong model); FrugalGPT/RouteLLM (dynamic routing
  pays only on high-volume homogeneous request streams — neither surface here qualifies). Token
  economy in this framework is already handled as targeted point fixes (v0.8.0 routing +
  quiet-on-success hooks + CLAUDE.md size budget + ADR hot-path split; watch list B-26).
  **Epistemic status**: haiku-tier adequacy has been *assumed* since v0.8.0 — the CHANGELOG's
  "without losing security or bootstrap quality" claim was never evaluated (evals have never
  gated a release, B-23). B-29 files the evidence gap. **Copilot surface (implementation fact,
  not a capability claim)**: tiering is currently wired on the Claude Code side only; the shipped
  `.github/agents/*.agent.md` wrappers deliberately pin no `model:`. GitHub documents a `model`
  frontmatter property for IDE custom-agent profiles (CLI honoring tracked upstream) — unverified
  here, so tiering must never become load-bearing for correctness on either surface.
- **Alternatives**: dynamic router / token-budget framework — rejected (over-engineering; review
  rigor is the product, and a cheap-model quality regression costs more than the tokens saved;
  no request volume on the meta layer). Shipped consumer-facing cost-knobs doc (how `/review`
  agents are tiered, how to change it, per-surface cost implications) — **deferred, not
  rejected**: it is a shipped doc change (lockstep + CHANGELOG + version bump); revisit on
  consumer cost feedback. One-line B-26 watch-list addition instead of an ADR — rejected: the
  question was asked directly and deserves a citable record. Downgrading more agents to haiku —
  rejected (fails the v0.8.0 rubric's stakes test).
- **Consequences**: future agents get tiered by the v0.8.0 rubric, cited from here. B-29 (haiku
  adequacy evals) and B-30 (`test-critic` missing from both repos' ARCHITECTURE.md §5 agents
  table — shipped doc drift found during this investigation) are filed. Meta layer note: it
  already routes where it matters (`tests/evals/run_evals.py` pins the grader to Haiku;
  adversarial critique passes run as subagents per WSD-010) — no further mechanism.

## WSD-015: Monorepo dist composition — concat-by-default + authored override + collision error
- **Date**: 2026-07-10
- **Status**: Accepted (implemented — B-25-EXEC Phase 4)
- **Decision**: `dist/monorepo` composes with two rules on top of the Phase-3 composer, keeping it
  dumb (copy + marker substitution + overlay only):
  1. **Marker resolution order**: `src/stacks/monorepo/snippets/<rel>/<NAME>` if present (authored
     merged/sectioned content) → else the dotnet snippet then the angular snippet, **raw
     concatenation** (union semantics; one-sided snippets pass through).
  2. **Whole-file overlay**: union of `dotnet/files` + `angular/files` + `monorepo/files`;
     `monorepo/files` wins; a path present in **both** stack dirs with **no** monorepo override is
     a **build error** — no silent last-wins, every collision is an explicit authored decision
     (red-tested: 38/38 collisions reported before any content existed).
- **Rejected**: (a) qualified include-markers (`@stack:dotnet:NAME`) inside monorepo snippets —
  a templating feature without enough consumers (Leanness #8); (b) composer-inserted
  `### .NET`/`### Angular` headings — the composer cannot know prose vs YAML-frontmatter vs script
  context and would corrupt single-line snippets.
- **Measured facts** (2026-07-10): all **114** shared snippet pairs were divergent (zero identical),
  so union-by-default only stays derived where content is additive-safe — **5** markers remain
  derived (audit-trail skip ×2 = independent guard branches; `route-prompt.sh` sensitive-grep =
  self-contained `if…fi` lines; 2 one-sided). **111 authored monorepo snippets + 38 authored
  whole-file overrides** carry everything else. `dist/monorepo` = **148 files** (128 common +
  10+10 stack-only). **D4 token gate: monorepo CLAUDE.md = 27,589 chars ≈ 1.17× the larger
  single-stack (angular 23,638) — well under the ~1.5× fallback trigger; single root file stands.**
- **Trap recorded** (see LEARNINGS): the plan's "snippets are additive-safe" held for the `.sh`
  twin but NOT its `.ps1` twin — `$sensitive = $lc -match 'A'` concatenated with `$sensitive =
  $lc -match 'B'` lets the second assignment overwrite the first (a .NET-only keyword would be
  erased). `route-prompt.ps1/sensitive-regex` needed an authored `(-match A) -or (-match B)`
  monorepo snippet. Additive-safety must be judged **per twin**, never assumed to transfer.
- **Accepted cost / discipline**: authored monorepo snippets/files are a third authoring surface
  for divergent content. Editing a stack snippet or stack whole-file that has a monorepo sibling
  requires reviewing that sibling in the same task (recorded in DEVELOPING.md). Core edits and the
  5 derived markers flow to all three dists automatically; CI dist-freshness still proves
  src→dist for all three.
